## Why V1.1

V1.0 shipped the entire runtime in Go. That gave us a single static binary,
but it forced every capability implementation (Agent Adapters, Tools, MCP
clients, Sandbox backends, Memory, Skill engine) to be re-implemented against
Go's stdlib ‚Äî fighting the AI ecosystem where Python is first-class.

V1.1 pivots the runtime along the language boundary:

- **Go** stays where it wins: the control plane (API, Scheduler, Worker
  Registry, DB, audit, observation aggregation, telemetry).
- **Python** takes the capability execution plane (Adapters, Tools, MCP,
  Sandbox, Memory, Skill engine, Session Runtime).

Communication is via a single wire contract at `proto/hnsx/v1/worker.proto`
plus the existing `observation.proto`. The Go server and the Python worker
both generate stubs from the same `.proto` source, so the contract cannot
drift.

## What `hnsx-worker/` is

A standalone Python 3.11+ package with its own `pyproject.toml`, venv, and
test suite. It is **not** an SDK; it's the worker's own internal source
tree. The Python stubs generated from proto live inside the package at
`hnsx_worker/proto/gen/hnsx/v1/` (gitignored, regenerated by
`make proto-py`).

The package depends only on:

- `grpcio` + `protobuf` ‚Äî wire contract
- `grpcio-tools` ‚Äî codegen for `make proto-py`
- `click` ‚Äî CLI entrypoint (`hnsx-worker --version`, `hnsx-worker check-proto`)

Adapter / Tool / Sandbox / Memory libraries (`anthropic`, `openai`, `httpx`,
`docker`, `redis`, ‚Ä¶) are **not** in `pyproject.toml` yet ‚Äî they are added in
Step 3 when the first real Adapter lands.

## Install

From the repo root:

```bash
make worker-install
```

This creates `hnsx-worker/.venv/`, installs `grpcio-tools` (for codegen),
and installs the package in editable mode with dev extras.

## Use

```bash
# Regenerate proto stubs (after editing proto/hnsx/v1/*.proto)
make proto-py            # Python only
make proto-all           # Go + Python

# Smoke check the generated stubs
hnsx-worker check-proto

# Run the pytest suite
make worker-test
```

End-to-end CLI usage (when the worker parent process is implemented in
Step 2):

```bash
# Future: hnsx-worker run --server=127.0.0.1:50051 --worker-id=w-1
# Future: hnsx-worker drain --server=127.0.0.1:50051 --worker-id=w-1
```
